# go-acp-server 使用说明

本文档介绍如何在 Go 项目中使用 `go-acp-server`，并说明它与模型运行时的职责边界。

## 1. 模块定位

`go-acp-server` 是一个“协议层 + 会话层 + 工具桥接层”模块，负责：

- 处理 ACP 的 JSON-RPC 协议消息
- 管理 session 生命周期（`session/new`、`session/load`、`session/list`、`session/prompt`、`session/cancel`）
- 输出 `session/update` 流式事件（含 `user_message_chunk`、`agent_message_chunk`、`agent_thought_chunk`、`plan`、`available_commands_update`、`current_mode_update`、`config_option_update`、`tool_call`、`tool_call_update`、`session_info_update`）
- 将 agent 侧工具调用转发到 ACP client（`fs/*`、`terminal/*`）

它不直接绑定某个特定模型框架。模型推理逻辑由业务侧通过 `PromptExecutor` 提供。

## 2. 核心接口

## `PromptExecutor`

你需要实现下面接口，并在里面完成“模型推理 + 可选工具调用循环”：

```go
type PromptExecutor interface {
    StreamReply(
        ctx context.Context,
        history []ChatTurn,
        prompt string,
        tools RuntimeToolInvoker,
        onChunk func(chunk string) error,
    ) (string, error)
}
```

参数说明：

- `history`: 当前会话历史
- `prompt`: 本轮用户输入（已做文本提取/标准化）
- `tools`: ACP 工具调用入口（调用会真正发给 client）
- `onChunk`: 流式输出回调（会发送 `session/update.agent_message_chunk`）

## `PromptExecutorWithUpdates`（可选）

如果你需要输出更丰富的 ACP 事件（例如 `agent_thought_chunk`、`plan`、`current_mode_update` 等），可实现扩展接口：

```go
type PromptExecutorWithUpdates interface {
    PromptExecutor
    StreamReplyWithUpdates(
        ctx context.Context,
        history []ChatTurn,
        prompt string,
        tools RuntimeToolInvoker,
        updates PromptUpdateWriter,
    ) (string, error)
}
```

`updates` 支持直接输出各类 session/update 事件，且向后兼容普通 `PromptExecutor`。

此外，默认 `session/prompt` 流程会自动发送一组“会话上下文事件”：

- `current_mode_update`：当前模式（默认 `default`）
- `available_commands_update`：由当前 runtime tools 动态生成
- `config_option_update`：由 session `cwd`、绑定的 `mcpServers` 与 client capabilities 动态生成

## `RuntimeToolInvoker`

```go
type RuntimeToolInvoker interface {
    ToolInfos(ctx context.Context) ([]RuntimeToolInfo, error)
    InvokeTool(ctx context.Context, toolCallID, toolName, argumentsInJSON string) (string, error)
}
```

- `ToolInfos`: 返回当前可用工具定义（由 client capabilities 决定）
- `InvokeTool`: 真实发起 ACP client 方法调用并返回结果

## 3. 快速接入

## 3.1 启动 server

```go
srv := acpserver.NewServer(executor, os.Stderr, "v0.1.0")
err := srv.Serve(context.Background(), os.Stdin, os.Stdout)
```

## 3.2 在执行器里接工具

典型模式：

1. 模型拿到 `ToolInfos`
2. 模型输出 tool call
3. 对每个 tool call 调用 `InvokeTool`
4. 把工具结果回填给模型继续推理
5. 产出最终文本并通过 `onChunk` 输出

如果你使用 Eino，可在你的 runtime 层把 `RuntimeToolInfo` 映射为 `schema.ToolInfo`，然后走模型的 tool-calling 能力。

## 4. 已支持的 client 工具方法

目前桥接实现支持：

- `fs/read_text_file`
- `fs/write_text_file`
- `terminal/create`
- `terminal/output`
- `terminal/wait_for_exit`
- `terminal/kill`
- `terminal/release`

是否暴露由 `initialize.params.clientCapabilities` 控制。

## 5. 如何确认“工具真的被调用了”

建议 `NewServer` 的 logger 传 `os.Stderr`，然后看这些日志：

- `[tool] start ...`
- `[acp->client] method=fs/read_text_file ...`
- `[client->acp] result=...`
- `[tool] done ...`

出现以上链路，说明不是“仅提示词提及工具”，而是发生了真实 ACP client RPC 调用。

## 6. 常见问题

## Q1: 为什么模型看到了工具，却没有调用？

这通常是模型策略问题，不是协议问题。你可以：

- 优化系统提示词，明确“需要时必须用工具”
- 在模型层设置更强的 tool-choice 策略（如果框架支持）

## Q2: `mcpServers` 是不是等于可直接调用的 ACP client tools？

不是。`mcpServers` 是会话上下文里提供给 agent 的 MCP 服务器配置；  
ACP 直接可由 agent 请求的 client 方法是 `fs/*`、`terminal/*` 这些。

## Q3: 如何验证模块本身？

```bash
cd crates/go-acp-server
go test ./...
```

模块内已有测试覆盖：

- 协议握手
- session/prompt 流式更新
- client tools 调用链路
- 工具日志输出
